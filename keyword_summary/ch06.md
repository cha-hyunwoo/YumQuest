- JPA란?

  JPA는 자바에서객체와 관계형 데이터베이스를 매핑하기 위한 ORM 표준 인터페이스이다.

  **등장 이유**

    - 자바는 객체 지향 언어이고, DB는 테이블 기반 구조를 사용하기 때문에 둘 사이에는 패러다임 불일치 문제가 발생한다.

  → SQL을 직접 작성하기 보다 자바 객체를 통해 데이터를 저장, 조회, 수정, 삭제 할 수 있게 도와주는 기술임



- N+1 문제란?

  데이터를 조회할 때 처음 1번의 쿼리 이후, 연관된 데이터 개수(N)만큼 추가 쿼리가 발생하는 문제이다.

  예를 들어 회원 10명을 조회했는데, 각 회원의 주문 정보를 조회해야 한다고 가정하면:

    ```sql
    회원 조회1번
    주문 조회10번
    ```

  총 11번의 쿼리가 발생한다

  이로 인한 문제점

    - 성능 저하
    - 응답 속도 저하

  해결방법

    - Fetch Join
    - EntityGraph
    - DTO 조회


- 지연로딩과 즉시로딩의 차이는?

  연관 객체를 언제 조회할 것인지 결정하는 전략

  **지연 로딩**

  실제 사용할 때 조회

    ```java
    @ManyToOne(fetch=FetchType.LAZY)
    ```

  처음에는 프록시 객체만 넣어두고, 실제 접근하는 순간 SQL이 실행된다.

  장점

    - 필요한 데이터만 조회
    - 성능 최적화 가능
    - 메모리 절약

  단점

    - N+1 문제 가능
    - 트랜잭션 밖에서 접근 시 예외 발생 가능

  **즉시 로딩**

  조회할 때 연관 객체도 함께 조회

    ```java
    @ManyToOne(fetch=FetchType.EAGER)
    ```

  장점

    - 바로 사용 가능
    - 코드 이해가 쉬움

  단점

    - 불필요한 Join 발생
    - 성능 예측 어려움



- JPQL란?

  JPA에서 사용하는 객체 지향 쿼리 언어이다.

  SQL은 테이블 기준

    ```sql
    SELECT * FROM member
    ```

  JPQL은 Entity 기준

    ```java
    SELECT m FROM Member m
    ```

  특징

    - 객체 중심 쿼리
    - DB  독립성 확보
    - 내부적으로 SQL로 변환됨

  장점

    - 객체지향적 코드 작성 가능
    - DB 변경 영향 적음

  단점

    - 복잡한 통계성 쿼리는 불편할 수 있음



- Fetch Join란?

  JPQL에서 연관 엔티티를 한 번의 SQL로 조회하는 기능이다.

  예

    ```java
    SELECT m FROM Member m
    JOIN FETCH m.orders
    ```

  사용 이유

  N+1 문제 해결

  장점

    - 쿼리 수 감소
    - 성능 향상

  단점

    - 컬렉션 fetch join 시 데이터 중복 가능
    - 페이징 제한 존재



- @EntityGraph란?

  `EntityGraph` 는 연관 엔티티를 함께 조회하도록 설정하는 기능이다.

  예

    ```java
    @EntityGraph(attributePaths={"orders"})
    List<Member> findAll();
    ```

  JPQL 없이도 Fetch Join과 비슷한 효과를 낼 수 있다.

  장점

    - 코드가 간결함
    - Repository 메서드와 자연스럽게 사용 가능

  단점

    - 복잡한 조건 표현 어려움



- commit과 flush 차이점은?

  **flush()**

  영속성 컨텍스트의 변경 내용을 DB에 SQL로 반영한다.

    ```java
    entityManager.flush();
    ```

  하지만 트랜잭션은 아직 안끝남

  SQL 실행 = O

  최종 저장 = X

  **commit()**

  트랜잭션을 최종 확정한다.

    ```java
    trasaction.commit();
    ```

  commit 전에 내부적으로 flush가 먼저 실행된다.

  SQL 실행 = O

  최종 저장 = O
