- 빌더패턴이란?

  복잡한 객체의 생성 과정을 단순화하기 위한 디자인 패턴

  **사용하는 이유**

  → 필드가 많아질수록 생성자 파라미터 순서가 헷갈리고 가독성이 떨어지는 문제 해결.

  **주요 특징**

    - 가독성:  `.name("현우").email("@")` 처럼 어떤 필드에 어떤 값이 들어가는지 명확히 보임
    - 유연성: 매개변수 순서에 얽매이지 않고 객체를 생성할 수 있음
    - 롬복(@Builder): 직접 빌더 클래스를 구현할 필요 없이 어노테이션 하나로 자동 생성 가능


- record vs static class

  **1. record (자바 16+ 신기술)**

  특징: `Getter`, `생성자`, `equals`, `hashCode`, `toString` 을 자바가 자동으로 만들어줌.



강점:

    - 코드 최소화
    - 불변성: 모든 필드가 자동으로 private final 이 됨

단점: 데이터 수정 불가(새로 만들어야 함)

    - 특정 라이브러리(QueryDSL 등)와의 호완성 문제 가능성



**2. static class (전통적인 방식)**

특징: 흔히 아는 클래스 선언 방식이며, 주로 롬복(`@Getter` , `@Builder` )과 함께 사용.

강점:

    - 유연성: 필요하면 데이터를 수정할 수 있음
    - 호환성: 거의 모든 자바 라이브러리에서 문제없이 작동

단점:

    - 보일러 플레이트: 어노테이션을 주렁주렁 달거나 코드가 길어질 수 있음


- 제네릭이란?

  클래스나 메서드에서 사용할 내부 데이터 타입을 컴파일 시가 아닌, 객체를 생성할 때 결정하도록 하는 기술



사용 이유

    - 중복 코드 방지: 똑같은 구조인데 타입만 다른 경우, 하나로 해결
    - 타입 안정성: 아무거나 다 들어오는 것이 아니라, 내가 정한 타입만 들어오라고 제한할 수 있어 런타임 에러를 방지(설계할 땐 하나로 사용할 땐 정한 타입으로)

주요 특징

    - <T>: 보통 Type 의 약자인 T 를 많이 씀
    - 유연한 바구니: 객체를 만들 때 <String> 을 넣으면 글자 바구니가 되고, <MemberDTO> 를 넣으면 멤버 정보 바구니가 됨



- @RestControllerAdvice이란?

  모든 Controller에서 발생하는 에러를 한곳에서 가로채 처리하는 전역 예외 처리기



사용 이유

    - 중복 제거: 컨트롤러마다 try-catch 를 사용하는 것을 방지
    - 응답 통일: 에러가 나도 ApiResponse 규격에 맞춰서 일관되게 대답 가능

주요 특징

    - @ControllerAdvice + @ResponseBody가 합쳐진 어노테이션
    - @ExceptionHandler와 함께 사용하여 예외별 처리 가능


- Optional이란?

  값이 있을 수도, 없을 수도 있음을 명시적으로 표현하는 컨테이너 객체



사용 이유

    - null을 직접 다루지 않기 위해
    - 반환값이 없을 수 있다는 사실을 명시적으로 표현

주요 특징 및 메서드

**값 꺼내기**

    - .get()
        - 값이 있을 때만 사용
        → 비어있으면 터짐(주의)


**값이 없을 때 처리**

    - .orElse()
        - 값 없으면 기본값 반환
        - 단점: 값이 있어도 기본값을 미리 생성함
    - .orElseGet
        - 값 없을 때만 함수 실행해서 생성
    - .orElseThrow()
        - 값 없으면 예외 발생

**값이 있을 때만 처리**

    - ifPresent()
        - 값 있으면 실행
    - .map()
        - 값이 있을 때만 변환
